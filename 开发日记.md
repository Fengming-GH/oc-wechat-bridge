# 开发日记

## 2026-06-25

### 1. 插件加载失败 — 三连 Bug

**根因1：少分号**

- `src/index.ts:274`：`await updateSessionIcon(client, existing, "normal")    return existing` 写在同一行，缺分号
- V8 通过 ASI 能运行，Bun 解析器拒绝 → 文件不加载
- 教训：`node --check` 不是可靠语法检查，V8 ≠ Bun 解析器

**根因2：缺依赖**

- `@opencode-ai/plugin` 不在 `node_modules`，OC 1.17.10 不会自动装
- `.opencode/package.json` 声明了依赖，但 OC 在项目插件加载时没自动 `bun install`
- 最终手动 `npm install @opencode-ai/plugin`

**根因3：解构字段名不匹配**

- `listAllSessions()` 返回 `{ flat, dirMap: dm }`，属性名是 `dirMap`
- `handleCommand` 里写成 `const { flat, dm } = ...` → `dm` 永远是 `undefined`
- `formatDirSessions(flat, dm, ...)` 收到 `undefined` → `dm.get()` 抛错 → "获取失败"
- 修复：`const { flat, dirMap: dm } = ...`

### 4. 用户取消被误判为错误

- `message.updated` 事件里 `info.error` 无法区分取消和真正错误
- 用户按 Escape/Ctrl+C 时，OC 返回 `MessageAbortedError`
- 插件无条件检测 `info.error` → 自动注入"检测到错误，继续工作"
- 修复：引入 `CONTINUE_ERRORS` 白名单，只对 `MessageOutputLengthError`、`APIError`、`UnknownError` 触发
- 参考：`oc-auto-continue` 已有相同逻辑

### 5. session.list({ directory }) 的坑

- `directory` 参数被 OC 忽略，所有调用返回当前项目会话
- 多目录场景下需对每个 `_projectDirs` 分别调 `session.list()` 再合并

### 6. context_token 过期问题

- 多步 AI 回复期间 token 变 stale
- `sendText` 发现 "Context token stale" 错误后删除缓存、静默丢弃
- 后续消息使用新 token
- 根源在 WeChat API 设计：每个 `sendmessage` 需要最新 `context_token`

### 7. @im.wechat 后缀一致性

**问题**：`saveState` 过滤掉 `@im.wechat` 后缀，只存裸 key → `loadState` 恢复裸 key → 运行时 `wechatSid.get(senderId)`（senderId 带后缀）永远不匹配 → unbind 跳过删除、`getOrCreateSession` 创建重复 session。

**修法**：`saveState` 改为**只保留** `@im.wechat` 后缀的 key：

```ts
// 改前（过滤掉后缀）
for (const [k, v] of contextTokens) { if (!k.endsWith("@im.wechat")) ct[k] = v }
// 改后（只保留后缀）
for (const [k, v] of contextTokens) { if (k.endsWith("@im.wechat")) ct[k] = v }
```

**教训**：微信 API 的 `from_user_id` 带 `@im.wechat` 后缀，所有存储和查询必须用统一格式。`sendmessage` 的 `to_user_id` 也需要带后缀才能发送成功（否则返回 `ret=-3`）。不要在持久化和运行时之间切换 key 格式。

### 8. sidTitle.has(existing) 竞态

**问题**：`getOrCreateSession` 用 `sidTitle.has(existing)` 判断 session 是否有效。但 `initSessionCache` 用 `setTimeout(0)` 推迟填充 sidTitle → 重启后第一条微信消息到达时 sidTitle 为空 → 条件不满足 → 创建新 session → 覆盖 `state.json` 的正确映射。

**修法**：移除 `sidTitle.has(existing)` 检查，直接信任 `wechatSid` 的映射：

```ts
// 改前
if (existing && sidTitle.has(existing)) {
// 改后
if (existing) {
```

**教训**：sidTitle 是内存缓存，不可作为会话存在性的判断依据。`wechatSid`（来自 state.json）才是权威来源。

### 9. session.updated 不应自动恢复图标

**问题**：在 `session.updated` handler 里加 `findWechatSender(s.id) → updateSessionIcon` 逻辑，导致 AI 执行工具触发事件时，给非绑定会话错误添加 📱 图标。

**修法**：恢复 `session.updated` 为纯 `sidTitle.set(s.id, s.title)`，不做图标恢复。图标只由 `updateSessionIcon` 在明确的业务路径（switch、idle、processInboundMessage）中管理。

**教训**：事件 handler 应只维护状态缓存，不应触发副作用。把多个职责塞进一个 handler 会制造难以诊断的跨域 bug。
