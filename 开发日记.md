### 7. @im.wechat 后缀一致性

**问题**：`saveState` 过滤掉 `@im.wechat` 后缀，只存裸 key → `loadState` 恢复裸 key → 运行时 `wechatSid.get(senderId)`（senderId 带后缀）永远不匹配 → unbind 跳过删除、`getOrCreateSession` 创建重复 session。

**修法**：`saveState` 改为**只保留** `@im.wechat` 后缀的 key：

```ts
// 改前（过滤掉后缀）
for (const [k, v] of contextTokens) { if (!k.endsWith("@im.wechat")) ct[k] = v }
// 改后（只保留后缀）
for (const [k, v] of contextTokens) { if (k.endsWith("@im.wechat")) ct[k] = v }
```

**教训**：微信 API 的 `from_user_id` 带 `@im.wechat` 后缀，所有存储和查询必须用统一格式。`sendmessage` 的 `to_user_id` 也需要带后缀才能发送成功（否则返回 `ret=-3`）。不要在持久化和运行时之间切换 key 格式。

### 8. sidTitle.has(existing) 竞态

**问题**：`getOrCreateSession` 用 `sidTitle.has(existing)` 判断 session 是否有效。但 `initSessionCache` 用 `setTimeout(0)` 推迟填充 sidTitle → 重启后第一条微信消息到达时 sidTitle 为空 → 条件不满足 → 创建新 session → 覆盖 `state.json` 的正确映射。

**修法**：移除 `sidTitle.has(existing)` 检查，直接信任 `wechatSid` 的映射：

```ts
// 改前
if (existing && sidTitle.has(existing)) {
// 改后
if (existing) {
```

**教训**：sidTitle 是内存缓存，不可作为会话存在性的判断依据。`wechatSid`（来自 state.json）才是权威来源。

### 9. session.updated 不应自动恢复图标

**问题**：在 `session.updated` handler 里加 `findWechatSender(s.id) → updateSessionIcon` 逻辑，导致 AI 执行工具触发事件时，给非绑定会话错误添加 📱 图标。

**修法**：恢复 `session.updated` 为纯 `sidTitle.set(s.id, s.title)`，不做图标恢复。图标只由 `updateSessionIcon` 在明确的业务路径（switch、idle、processInboundMessage）中管理。

**教训**：事件 handler 应只维护状态缓存，不应触发副作用。把多个职责塞进一个 handler 会制造难以诊断的跨域 bug。

### 11. code review 误判澄清

2026-06-27 对 `oc-wechat-bridge/src/index.ts` 做代码审查后，有以下设计意图需要记录，避免后续分析人员误判。

#### 11a. handleFirstContact — 预绑定引导，不是丢消息

**误解**：新用户第一条消息被引导语"吃掉"，没有进入 AI，是设计缺陷。

**设计意图**：这是**故意的预绑定引导流程**，不是 bug。

```ts
// L488-489 processInboundMessage
if (!wechatSid.has(senderId)) {
  if (await handleFirstContact(...)) return  // ← 发引导列表，不送 AI
}
// 下面的 getOrCreateSession + promptAsync 不会执行
```

流程：
1. 新用户发消息 → `handleFirstContact` 返回 `/switch N` 的会话列表
2. 用户手动选 `/switch 2`（或 `/new`）→ 绑定到 wechatSid
3. 后续消息才进入 `getOrCreateSession` → `promptAsync` → AI 处理

**理由**：插件不知道新用户想用哪个 OC会话（多项目目录场景），强制自动创建会覆盖用户预期。先给列表让用户选，再开始对话。

#### 11b. wechatSid — N:1 映射，不是内存泄漏

**误解**：`wechatSid.delete` 只在 session.deleted 和 unbind 时发生，有内存泄漏风险。

**设计意图**：**一个微信 senderId 最多绑定 1 个 OC session**，映射是 **N:1**（N个微信联系人 → 各自最多1个 session）。不是 1:N。

- `wechatSid` 大小 = 有映射的微信联系人数量，通常 < 100
- 即使完全不清理，内存占用也极小（几 KB）
- `state.json` 在每次写（session 创建/切换）时整体覆盖，不会无限膨胀
- session.deleted 事件会清理对应映射，`/unbind` 也会删

真正需要关心的是 `wechatSid` 的**准确性**（删除的 session 是否清理了映射），而不是"泄漏"。

#### 11c. session.updated 中 updateSessionIcon 无 await — 故意的 fire-and-forget

**误解**：`session.updated` handler 里调 `updateSessionIcon` 没加 await，可能丢错误。

**设计意图**：这是故意的 fire-and-forget 模式，原因：

1. 事件处理器是同步的，OC SDK 不等待 Promise
2. 图标更新是 cosmetic，失败不影响功能（`sidTitle` 缓存已更新）
3. 内部已有 try-catch，`.catch()` 是第二层保险
4. 下次 `session.idle`、`getOrCreateSession`、`/switch` 等路径会重新修复图标

**代价**：网络抖动导致一次图表更新静默失败时，图标在下次事件到来前保持错误状态。可接受。

### 12. CLI-WeChat-Bridge 审批系统分析

**来源**：`UNLINEARITY/CLI-WeChat-Bridge`（v1.1.1，387 stars）
**分析范围**：仅审批（approval/permission）系统，不涉及其 daemon/multi-adapter 架构。

#### 12a. 完整流程

```
OC                  OpenCodeAdapter          BridgeController         WeChat
 │                       │                        │                    │
 │  permission.asked     │                        │                    │
 ├──────────────────────► │                        │                    │
 │                       │  approval_required      │                    │
 │                       ├───────────────────────► │                    │
 │                       │                        │  formatApprovalMsg │
 │                       │                        ├───────────────────► │
 │                       │                        │  "OC 请求权限：...   │
 │                       │                        │  回复 /confirm      │
 │                       │                        │  或 /deny"         │
 │                       │                        │                    │
 │                       │                        │  ← "同意"          │
 │                       │                        │  ← "/confirm"      │
 │                       │                        │  ← "yes"           │
 │                       │                        │                    │
 │                       │  resolveApproval()     │                    │
 │                       │ ◄────────────────────── │                    │
 │  permission.respond() │                        │                    │
 │ ◄─────────────────────┤                        │                    │
 │                       │                        │                    │
 │  resolved             │                        │                    │
 ├────► OC 继续执行       │                        │                    │
```

#### 12b. 关键设计决策

| 要素 | CLI-WeChat-Bridge 做法 | 备注 |
|------|----------------------|------|
| **触发** | adapter 监听 `permission.asked` / `permission.updated` 事件 | 我们的 `permission.ask` handler 就是入口 |
| **提示格式** | `formatApprovalMessage()` 生成结构化文本发给微信 | 含 adapter、summary、target 字段 |
| **解析回复** | 双模式：`/confirm` 命令（精确匹配）和纯文本 `"yes"` / `"同意"`（模糊匹配） | 微信用户不需要记命令 |
| **执行** | `adapter.resolveApproval()` → `client.permission.respond({ response: "once" | "reject" })` | OC SDK 接口明确 |
| **安全限制** | 自动拒绝写本地文件的审批（WeChat 不能直接写文件） | 可选，视需求决定 |
| **超时** | stateStore 记录 pending + 心跳提醒 | 可简化为一小时后自动拒绝 |

#### 12c. 关键代码定位（`bridge-adapters.opencode.ts`）

```
事件捕获:   L1012-1016  permission.updated / permission.asked
构建审批:   L1288-1330  buildPendingPermission() → OpenCodePendingPermission
发送到微信:  L1187-1199  BridgeController: "approval_required" → formatApprovalMessage → queueWechatMessage
解析回复:   bridge-utils.ts:L282-287  (/confirm /deny) 和 L317-326 (yes/no)
执行审批:   L1503-1543  BridgeController: case "confirm"/"deny" → adapter.resolveApproval()
OC SDK 调用: L456-488   resolveApproval() → client.permission.respond()
自动拒绝:   L1216-1247  getWechatOutboundAttachmentPermissionDenyMessage()
```

#### 12d. 对我们的参考价值

1. **Promise 挂起**模式已验证可行：`permission.ask` handler 不立即返回，而是 await 一个 Promise，由微信回复触发 resolve
2. **纯文本匹配**（`"同意"` / `"yes"`）降低了微信用户的使用门槛
3. 我们已有的 `permission.ask` handler（返回 `"ask"`）可直接替换为挂起模式
4. CLI-WeChat-Bridge 的代码偏重（daemon + 多 adapter），可以直接借鉴其核心 Promise 逻辑，不搬整体架构
